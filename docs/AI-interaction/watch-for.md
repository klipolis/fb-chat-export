# What to Watch for Going Forward

- **Co-located duplication**: adjacent files in `src/shared/` can contain copy-pasted logic without anyone noticing during normal development. A structured deduplication review for each new feature flag would help.
- **Behavioral drift between server and frontend**: when the same concept (alias replacement, URL stripping, content classification) has independent implementations, they develop different edge-case behavior. Shared code in `src/shared/` should be the default, not the exception.
- **Error handling normalization**: a single bare `catch {}` in one module makes it the local convention. Review all `catch` blocks as a batch when touching a file.
- **Validation gaps from hardcoded generation**: any file produced by `build-server.cjs` that isn't declared in `export-config.json` is invisible to schema validation. Cross-check after every server change.
- **Worker thread error propagation**: parallel workers communicate errors via `parentPort.postMessage({error})`. Ensure every consumer of `processFilesInParallel` checks for error responses and rejects the pool promise, not just worker `exit` codes.
- **Cache manifest staleness**: the build cache at `data-output-auto/build-cache.json` records input file states. If a file is deleted from `data-input-test/`, the next rebuild must detect it via the cache and remove stale output. If the cache file itself is missing or corrupted, the build must fall back to a full rebuild.
- **Sensitive sender names in logs and docs**: Sender names from chat data are modifiable inputs and must not appear verbatim in logs, changelogs, TODO files, or documentation. Describe them generically ("a detected sender", "an alias target") to avoid leaking real names into project artifacts.
- **UTF-8 encoding corruption in generated files**: The build pipeline reads/writes all files as UTF-8, but editors or git operations on Windows can add a BOM or corrupt multi-byte characters. After a full rebuild, verify that non-ASCII characters (Hungarian text, em dashes, pound signs) survived intact. If corruption appears, restore from git and check the source files in `data-input-test/` for encoding issues.
- **CRLF/LF mismatch in generated output**: `data-output-auto/` files should use LF endings (build enforces this with `.replace(/\r\n/g, '\n')`). The `.editorconfig` and `.gitattributes` now explicitly handle this, but any new build step that writes to `data-output-auto/` must also normalize to LF.
